Flipper App Packages (FAPs) are `.elf` executables that run from the SD card. Because the MCU cannot execute code directly from external storage, the App Loader copies each FAP to RAM, processes relocations, and resolves imported symbols against the firmware's exported symbol table before starting the app.

Compatibility between a FAP and the firmware it runs on is determined by the **API version** embedded in the FAP's metadata.

## API version format

API versions follow semantic versioning with a major and minor number (`major.minor`).

| Component | Incremented when |
| --------- | ---------------- |
| Major     | There are breaking changes to the API (see below) |
| Minor     | New symbols are added without removing or changing existing ones |

The App Loader checks that the FAP's **major** API version matches the firmware's major API version. A mismatch prevents the app from running.

## Breaking changes

The following changes are considered breaking and require a major version increment:

- **Removing** a function or a global variable from the API
- **Changing the signature** of an exported function

Adding new symbols (functions, variables) is non-breaking and increments only the minor version.

## The api_symbols.csv file

The set of symbols available to external apps is defined in `api_symbols.csv`, located in the hardware target's directory under `targets/`. Each row in the file represents one exported symbol.

Each entry carries one of three marks:

| Mark | Meaning |
| ---- | ------- |
| `+`  | Symbol is exported and available to FAPs |
| `-`  | Symbol is excluded from the FAP API |
| `?`  | Symbol is newly detected and awaits a decision |

## How fbt manages API versions

API versioning is mostly automated. When you rebuild the firmware, `fbt` compares the symbols exposed in `SDK_HEADERS` against the current `api_symbols.csv`. If new symbols are detected:

<Steps>
  <Step title="Build stops">
    `fbt` halts the build and updates `api_symbols.csv`, marking each new symbol with `?`.
  </Step>
  <Step title="Review the changes">
    Open `api_symbols.csv` and change each `?` entry to `+` (expose to FAPs) or `-` (exclude from the API).
  </Step>
  <Step title="Rebuild">
    Run `fbt` again. The build proceeds only when no `?` entries remain.
  </Step>
</Steps>

<Warning>
`fbt` will not build the firmware while any `?` entries remain in `api_symbols.csv`. Every new symbol must be explicitly marked `+` or `-`.
</Warning>

## When to manually increment the major version

`fbt` manages the API version automatically in all normal cases. The **only** situation that requires a manual major version increment is when you change an existing `+` entry to `-` — that is, when you remove a previously exported symbol from the public API.

<Note>
Do not manually increment the major version for any other reason. Adding new symbols or changing unexported symbols does not require a version bump and should be handled by `fbt` automatically.
</Note>

## The symbol table

The symbol table is a compiled list of all `+` symbols and their addresses in the firmware. It is generated by `fbt` from `api_symbols.csv` and is built as part of the `fap_loader` app.

When the App Loader runs a FAP, it uses the symbol table to resolve the addresses of all functions and variables the FAP imports. If any imported symbol is missing from the table:

- `fbt` issues a warning listing the unresolved symbols at build time.
- The app cannot run on the device until all required symbols are present in the symbol table.

This means an app that imports a symbol marked `-` (or not present in `api_symbols.csv` at all) will fail to load.
